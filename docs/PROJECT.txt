# Personal Economy Project – Project Definition (v1)

Repo name: personal-economy

---

## 1. Vision

Build a personal economy tool that gives a clear, honest picture of your finances and is realistic to maintain over time.

- Track income and expenses by category.
- See monthly surplus/deficit and how it evolves.
- Support weekly / monthly / yearly views (monthly as the default).
- Later: plug in investment context (high level) and more automation.

Multi‑user from day one (simple email/password login), but designed around **your** workflow first.

---

## 2. v1 Goals (Concrete)

### 2.1 What v1 should do

1. **User Accounts**
- Email/password signup/login/logout.
- Password reset flow (via Supabase auth).
- Each user’s data is isolated.

2. **Data Ingest**
- Manual upload of CSV / (optionally) PDF statements from:
- Sparebanken Norge (bank).
- Curve.
- Trumf.
- v1: focus on CSV uploads; PDFs can be an advanced path.

3. **Transaction Storage**
- Normalize all uploaded transactions into a unified `transactions` table:
- date, amount, currency
- merchant / description
- source (bank/card)
- category
- notes
- user_id
4. **Summaries & Views**
- Monthly view (default):
- total income, total expenses, net (surplus/deficit),
- spending by category (bar chart / list).
- Weekly view:
- weekly spending vs average.
- Yearly view:
- monthly totals across a year.
- Simple text summary for each month (“You spent X, saved Y, top categories were…”).

5. **Configurable Categories**
- Basic default categories:
- Housing, Groceries, Eating out, Transport, Subscriptions, Other.
- User can edit category assignments for specific transactions.
6. **Future (v2+) hooks**
- Simple fields for investment context (not full tracking yet):
- e.g. “total invested”, “monthly contribution”, “target”.
- Space for:
- alerts / anomalies (“subscriptions increased by X%”),
- personal goals (“save X/month”, “runway Y months”).

---

## 3. Data & Privacy

- Data lives in **Supabase Postgres**, under the user’s own Supabase project.
- No direct bank API scraping in v1:
- All data enters via **manual uploads** (CSV and possibly PDF).
- Sensitive secrets (DB URLs, service roles, Supabase keys):
- stored in `.env.local` for local dev,
- stored as secrets in GitHub / Cloudflare for deployment.

---

## 4. Tech Stack (v1)

### 4.1 Frontend

- **Framework**: Next.js (latest stable).
- **Styling**: Tailwind CSS.
- **Goal**: “Good enough” design that **epsilon** can later refine using `frontend-design`.

Core pages/components:
- `/auth` (login/register/reset) – or dedicated auth routes as per Supabase/Next integration.
- `/dashboard`
- High-level monthly overview (current month by default).
- Cards: income/expenses/surplus, category breakdown.
- `/transactions`
- Table of transactions with filters (date range, category, source).
- `/upload`
- CSV upload form (file selector, source type).
- Display basic stats post-upload.

### 4.2 Backend

- **Backend style**: Fullstack Next.js (no separate backend service initially).
- Next.js API routes or server actions will:
- handle CSV uploads,
- parse/normalize transactions,
- talk to Supabase.
- **Auth**: Supabase Auth (email/password).
- Signup, login, password reset.
- **Database**: Supabase Postgres.

### 4.3 Storage / Schema (initial idea)

Tables (simplified):

- `users`
- managed by Supabase auth.
- each has `id` that we use as foreign key.
- `accounts`
- `id`
- `user_id`
- `name` (e.g. “Sparebanken Norge Main”)
- `type` ("bank", "card", "other")
- `provider` ("sparebanken", "curve", "trumf", etc.)

- `transactions`
- `id`
- `user_id`
- `account_id`
- `date`
- `amount` (signed: positive income, negative expense)
- `currency`
- `description`
- `merchant`
- `category` (string; normalized list)
- `source_raw` (original source type/filename)
- `created_at`

- `summaries`
- `id`
- `user_id`
- `period_type` ("month", "week", "year")
- `period_start` (date)
- `period_end` (date)
- `total_income`
- `total_expenses`
- `net`
- `data` (JSONB for category breakdown and extra metrics)

- `settings` (optional)
- `user_id`
- `base_currency`
- `targets` (JSON; e.g. savings goals, runway targets)

### 4.4 Hosting

- **App**: Deployed on **Cloudflare Pages/Workers** (like your other projects).
- **DB/Auth**: Supabase (Postgres + Auth).

## 6. Agent Automation & Pipelines

### 6.1 Monthly review

For now: **on-demand** to save tokens.

- User triggers a review (button in UI or manual command).



- Backend:
- retrieves last month’s transactions for the user,
- aggregates to category totals and net,
- stores summary in `summaries`.
- beta:
- optionally generates a textual report:
- stored in `reports/YYYY-MM.md` in the repo,
- or in a `summary` field in DB.
- Later:
- We may set up an OpenClaw cron job to trigger this monthly.
- That would cause token usage even without changes, so we delay it until v1 is stable.

### 6.2 Weekly / yearly rollups

- Weekly:

- derived from transactions; useful as a “pulse” view.
- Yearly:
- derived from monthly summaries; useful for longer trend.

### 6.3 Output locations

- **In the webapp**:
- Dashboard panels and charts,
- Summary text for each period,
- Possibly a “Reports” section listing previous months.

- **Optional Discord integration** (later):
- A brief summary message in a channel (e.g. `#personal-economy`):
- “March 2026: income X, expenses Y, net Z, top categories A/B/C”.
